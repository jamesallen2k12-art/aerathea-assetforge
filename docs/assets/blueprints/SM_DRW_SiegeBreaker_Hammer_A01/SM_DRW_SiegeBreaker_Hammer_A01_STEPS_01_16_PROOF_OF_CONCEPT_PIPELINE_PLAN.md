placement, Unreal LOD/collision/material/performance validation, asset-library

## Artistic-Soul Keeper Features

The following features are mandatory review subjects throughout the pipeline:

1. Monumental, top-heavy Aerathean Dwarven great-hammer silhouette.
2. Two irregular faceted stone strike masses that are balanced without reading
   as bland mirrored boxes.
3. Dense, believable aged-bronze and dark-steel structural bracing.
4. A dominant layered central diamond shield/crystal focal point.
5. A narrow engraved shaft contrasted against the massive head.
6. Dark-brown crisscross leather grip and a weighted faceted pommel.
7. Cold blue rune light used sparingly against warm aged metal.
8. Authored 360-degree construction with no blank backs, copied facades,
   cards, decorative carriers, or projection shells.
9. Wear that tells the story of a forged and used siege relic without uniform
   procedural noise.
10. Large forms built as geometry, with cracks, grain, fine engraving, and
    micro-wear reserved for texture and Normal detail when appropriate.

A technical pass cannot approve artistic fidelity. Loss of any keeper feature
is an artistic-gate failure even if numeric, topology, export, or render audits
pass.

## Autonomous Repeatability Contract

### Required Operating Modes

The pipeline must support two explicit modes:

1. `first_proof_run`
   - consumes only approved immutable source and plan inputs;
   - executes every deterministic step without outside intervention after one
     explicit end-to-end execution authorization;
   - requests one batched Flamestrike decision only if Step 09 produces
     unresolved source-authority items that Step 10 cannot resolve from already
     approved rules;
   - creates all geometry, UVs, textures, materials, LODs, collision, exports,
     renders, and audits without manual Blender edits;
   - stops at the final Flamestrike artistic decision.
2. `locked_replay`
   - consumes the approved source, environment lock, formulas, interpretation
     decisions, artistic-soul contract, build recipe, and validation rules;
   - accepts no interactive production decisions;
   - completes from intake through Step 16 with zero outside involvement;
   - fails closed instead of prompting when any immutable input, environment,
     formula, dependency, or expected authority is missing or changed.

### Minimal Outside-Input Budget

The first proof run permits only:

1. one explicit authorization for the exact end-to-end plan;
2. at most one batched Step 10 source/interpretation decision if the source
   evidence produces unresolved blockers; and
3. one final Flamestrike artistic approval, rejection, or blocked decision.

Intermediate technical, geometry, material, and artistic-soul checks are
mandatory but must run automatically. Their proof packages remain available
for inspection, and any failed mandatory check stops the pipeline. They do not
require routine outside input when they pass under already approved rules.

Flamestrike retains final artistic authority. Autonomous completion may produce
a `DCC game-ready candidate`; it may not self-promote that candidate to visual
canon, approved library status, `Fully game-ready`, or any other status that
requires Flamestrike or Unreal approval.

### Deterministic Execution Requirements

- one manifest-driven entry command or local application action starts or
  resumes the complete run;
- the state machine writes every transition and refuses an illegal transition;
- tool versions, Blender version, Python/runtime dependencies, color-management
  configuration, render backend, device class, locale, units, seeds, thread
  behavior where relevant, and environment variables are locked;
- production requires no network access, package download, cloud generation,
  external service, or transient chat content;
- no manual mesh edit, sculpt, UV move, texture paint, material adjustment,
  export setting, or validation override is permitted;
- every variable production decision comes from an immutable input manifest or
  an approved formula;
- every command, parameter, input hash, output hash, tool version, state
  transition, failure, and retry is written to an immutable event trace;
- interrupted execution resumes from the last verified state without rerunning
  or silently changing approved earlier outputs;
- any stochastic operation uses recorded fixed seeds and must reproduce the
  same canonical result;
- any intrinsically nondeterministic or hardware-dependent method must be
  replaced, pinned to a proven deterministic path, or classified as blocked.

### Canonical Equality Requirements

Binary file hashes are required where the file format is deterministic. Where
container metadata may differ, equality must use a separately defined canonical
representation that removes only explicitly non-semantic metadata.

At minimum, locked replay must reproduce:

- exact source, crop, mask, scanline, formula, and decision-manifest hashes;
- exact canonical component IDs, vertex positions, face indices, normals, UVs,
  material assignments, transforms, pivot, bounds, sockets, and collision data;
- exact texture mip-0 pixel hashes and map metadata;
- exact LOD triangle counts and canonical geometry hashes;
- exact FBX/GLB clean-reimport canonical geometry/material results;
- exact fixed-camera binary silhouette masks and XOR counts;
- exact validation verdicts and state-transition trace;
- deterministic review-board composition and recorded render-environment
  identity.

Expected-value fields never substitute for observed replay results.

### Proof-Of-Concept Completion Test

Step 16 is not complete until all of the following pass:

1. `Run A` completes from immutable inputs in a fresh isolated output root with
   no manual production intervention.
2. `Run B` completes from the same immutable inputs in a second fresh isolated
   output root with no reuse of Run A production outputs.
3. Run A and Run B have the same state path and canonical results under the
   equality requirements above.
4. A forced interruption resumes correctly from the last verified state.
5. Separate corrupted crop, scale, mask, landmark, axis, source hash, and output
   hash tests fail at the correct gate and cannot advance.
6. No unrecorded file read, network access, prompt dependency, manual DCC edit,
   or hidden fallback occurs.
7. One complete rebuild recipe and reproducibility manifest can recreate the
   result without relying on the original agent or conversation.

Failure of any repeatability test leaves the pipeline `blocked` even when one
production candidate looks correct.

## Non-Deviation Enforcement Contract

The approved path is strictly sequential:

`01 -> 02 -> 03 -> 04 -> 05 -> 06 -> 07 -> 08 -> 09 -> 10 -> 11 -> 12 -> 13 -> 14 -> 15 -> 16`

For every step:

1. A versioned step contract must name the exact authority, inputs, actions,
   outputs, forbidden methods, validators, drift controls, and next decision.
   Flamestrike has waived routine display of the full contract documents;
   contract existence, rule completeness, and scope compliance remain mandatory
   and must be audited before execution.
2. Execution authority authorizes only that step's declared actions. When a new
   approval is required, Codex may present a concise exact scope summary instead
   of the full contract document.
3. Deterministic validation reports `pass` or `fail`; it does not grant
   approval.
4. Each output receives the exact machine and artifact disposition required by
   its contract. Flamestrike classifies only the batched unresolved-authority
   decision, when one is required, and the final artistic result. Under an
   explicitly approved end-to-end proof-run authority, all other intermediate
   steps auto-advance only when every deterministic gate passes and no new
   authority is required.
5. The next step remains locked until the prior output has the exact approved
   machine or Flamestrike disposition required by this plan and its restart-safe
   handoff is recorded.

The following are prohibited deviations:

- skipping, merging, reordering, or silently repeating a step;
- substituting a method, source, formula, tolerance, component scale, or
  validation rule;
- using an unapproved fallback after a tool or dependency failure;
- weakening a failed exact gate with IoU, visual similarity, average error,
  convenience tolerance, or a target value copied into an observed field;
- repairing forward from a failed, drifted, quarantined, or invalid output;
- treating technical completion as artistic approval;
- using an approval for adjacent or later work.

If an unplanned condition occurs, execution stops with one of:

- `Blueprint block: source authority missing`
- `Blueprint block: rule missing`
- `Blueprint block: approval missing`
- `Blueprint block: drift risk`

Continuation then requires an evidence-linked, versioned amendment approved by
Flamestrike. An approved amendment changes the authoritative plan version; it
is not a silent deviation. Without that amendment, affected output is
`invalid` or `quarantined` and cannot unlock the next state.

## Pipeline-Wide Technical Controls

- Preserve immutable source bytes and SHA-256 hashes.
- Decode losslessly and use integer source coordinates.
- Keep expected values separate from observed values.
- Permit only the declared uniform transform for each complete source view.
- Preserve explicit unknowns instead of inventing depth or hidden surfaces.
- Separate evidence, interpretation, candidate output, and proof-only output.
- Use independent builders and validators. A validator may not trust builder
  booleans, custom properties, target values, or the builder's own claims.
- Treat missing, skipped, unreadable, or unevaluated gates as failures.
- Keep measurement boards free of candidate fills, inferred contours, and
  geometry-like solution overlays.
- Open every required source, measurement, interpretation, geometry, material,
  and final review artifact automatically in a visible desktop window. Routine
  display of the step contract documents themselves is waived unless
  Flamestrike requests one or a contract conflict requires review.
- Checkpoint before and after long Blender, render, export, package, and
  validation operations and after failures.
- Stage, commit, and push only the exact approved dependency-complete scope.
- Perform a fresh resume handshake after mandatory resets and major Steps
  11-16.

If a new automated reconstruction method is required, it must first pass an
isolated known-object calibration test and separate corrupted crop, scale,
mask, landmark, and axis tests. A full desktop application is not authorized by
this plan; automation implementation requires its own approved contract.

Visible review artifacts must still be generated and opened when Core requires
them. In autonomous proof mode, successful intermediate review artifacts do not
require Flamestrike interaction unless the step reports a block or the plan
assigns a non-delegable final decision.
